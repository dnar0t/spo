## 27.2 — Точный формат поля оценки в YouTrack

**Тип поля:** `PeriodProjectCustomField`  
**Формат хранения:** целое число в **минутах** (поле `minutes` внутри `PeriodValue`).  
**Отображение (presentation):** строка вида `"50н"` (50 недель), `"33н 6ч 30м"` (33 недели 6 часов 30 минут).

Пример из задачи RD-1319:
```json
{
  "minutes": 120000,
  "presentation": "50н"
}
```
Конвертация: `120000 минут ÷ 60 = 2000 часов`.

**Рекомендация для СПО:** Запрашивать из API поле `customFields.value.minutes`, конвертировать в часы делением на 60.

## 27.3 — Настройка sprint/agile board и custom field спринта

В YouTrack обнаружены **два подхода** к управлению спринтами:

### Подход А: Agile/Sprints API

Существуют доски с именованными спринтами, имеющими даты начала и окончания. Ключевые доски:

| Доска | ID | Характеристика |
|---|---|---|
| Cloud | 128-96 | Спринты с датами: `Спринт 1 [03.02-02.03]`, ..., `Спринт 16 [01.04-30.04]` |
| Спринты по сотрудникам | 128-99 | Те же спринты, что и Cloud |
| Спринты по системам | 128-105 | Те же спринты, что и Cloud |
| R&D | 128-287 | Только "Первый спринт" (без дат) |
| Инфраструктура | 128-11 | Только "Первый спринт" |

### Подход Б: Custom field "Спринт" (Enum)

Enum-поле `"Спринт"` (id: 117-833) содержит **32 значения** — названия спринтов:
- `Спринт 1 [06.06-19.06]`, `Спринт 2 [20.06-03.07]`, ... (более старые, 2-недельные)
- `Спринт 1 [03.02-02.03]`, `Спринт 2 [03.03-30.03]`, ..., `Спринт 16 [01.04-30.04]` (более новые, календарно-месячные)
- `Рутина`, `План Пуля`, `тест` (специальные)

**Рекомендация для СПО:** Использовать **Agile/Sprints API** как основной источник данных о спринтах (для получения дат начала/окончания). При синхронизации задач дополнительно учитывать Enum-поле `"Спринт"` как fallback (если задача привязана к спринту через него).

При передаче плана в YouTrack (п. 6.5 ТЗ) задачи должны добавляться в спринт через Agile/Sprints API и/или заполнять Enum-поле `"Спринт"`.

## 27.4 — Источник для постановщика

В YouTrack присутствуют два источника:

1. **Reporter (встроенное поле задачи)** — стандартное поле YouTrack, одиночный пользователь. Доступно всегда.
2. **Custom field `"Постановщик"`** (id: 94-51) — тип `MultiUserIssueCustomField` (множественный выбор пользователей). Позволяет указать несколько постановщиков.

**Рекомендация для СПО:** Использовать Reporter как основной источник постановщика задачи. Custom field `"Постановщик"` (кастомный) использовать как расширенный/дополнительный — отображать, если он заполнен, иначе Reporter.

## 27.5 — Точные значения статусов, которые считаются завершёнными

В YouTrack завершённость определяется флагом `isResolved: true` у элемента State-бандла.

Основной бандл "Состояния" (id: 98-0) содержит следующие завершённые статусы:

| Статус | isResolved |
|---|---|
| `Завершено` | true |
| `Архив` | true |

В других бандлах (используемых в разных проектах) завершёнными также являются:
- `Done` (ЕВП)
- `Duplicate` (ЕВП)
- `Размещено в продакшн`
- `Передача ТП/ОТК`
- `Запрет на тест в прод` (специфический статус поставок)
- `Запрет распространения` (специфический статус поставок)
- `Распространен` (специфический статус поставок)

Полный список незавершённых статусов основного бандла:
- `Backlog`, `Test`, `Work`, `Готово к тестированию`, `На уточнении`, `Размещено в продакшн`, `Спринт бэклог`, `Запланированно`, `На проверке`, `Готово к деплою`, `Принято`, `Доработка`, `Перенесена`, `Бэклог спринта`

**Рекомендация для СПО:** Проверять `isResolved` у статуса задачи, а не сверять по имени. Это гарантирует корректную работу при изменении набора статусов в будущем.

## 27.6 — Точные значения приоритетов

**Фактические значения (EN):**

| Значение | Приоритет |
|---|---|
| `Blocker` | Наивысший |
| `High` | Высокий |
| `Medium` | Средний |
| `Low` | Низкий |

**Важное замечание:** В ТЗ версии 1.0 указаны русские названия `Блокер` и `Высокий`. Фактически в YouTrack используются **английские названия**. Необходимо скорректировать маппинг в ТЗ и в настройках системы.

Допустимо в интерфейсе СПО отображать русские локализованные названия, но при сопоставлении с полем YouTrack использовать английские значения.

## 27.7 — Доступность Hub как OAuth/OIDC-провайдера

**Результат:** Hub API недоступен.

При попытке обращения к эндпоинту `https://youtrack-dev.infomatix.tech/hub/api/rest/providers` получен ответ `404 Not Found`.

При попытке обращения к `https://youtrack-dev.infomatix.tech/hub/api/rest/users` с токеном, выпущенным для другого пользователя, получен ответ `401 invalid_token`.

**Вывод:** Hub не настроен как OAuth/OIDC-провайдер, либо используется версия YouTrack без Hub. Для аутентификации потребуется **прямой LDAP/LDAPS к Active Directory**.

## 27.8 — Налоговые формулы

**Решение:** Настройка налоговых формул вынесена на уровень роли **Бухгалтер**. Формулы (НДФЛ, страховые взносы, резерв отпускных и др.) являются редактируемыми через интерфейс администрирования. Бухгалтер самостоятельно определяет корректные проценты и правила расчёта. Никаких жёстко заданных формул на уровне кода не требуется — механизм настраиваемых процентов и преднастроенных формул (п. 14.4 ТЗ) покрывает эту потребность.

## Решения по уточняющим вопросам к ТЗ

В дополнение к исследованным вопросам пункта 27, в процессе анализа ТЗ были выявлены и согласованы с заказчиком следующие уточнения:

### Вопрос 1: Глубина иерархии задач в бэклоге

**Решение:**
- При формировании бэклога в СПО показываются задачи **не выше уровня "История"**.
- Если у задачи или ошибки (бага) нет родителя-Истории, она показывается в бэклоге как есть.
- Если у Истории есть родитель (вышестоящий эпик или фича), то при наведении на неё отображается **tooltip** с полным названием задачи и номером тикета родителя.

### Вопрос 2: Несколько исполнителей задачи (Multi-user Assignee)

**Решение:**
- При загрузке задач из YouTrack, если у задачи несколько исполнителей, необходимо при синхронизации **избавляться от нескольких исполнителей** — оставлять только одного.
- При выгрузке плана из СПО в YouTrack у задачи должен быть выставлен **один исполнитель** — тот, на кого запланирована задача.

### Вопрос 4: Поле "Подсистема"

**Решение:** Не используется. Система оперирует **только** признаком `"Система"`. Поле `"Подсистема"` (и другие вспомогательные поля) игнорируются.

### Вопрос 6: Задачи без оценки

**Решение:**
- Задачи выгружаются в СПО **независимо от того, заполнена в них оценка в YouTrack или нет**.
- СПО **не даёт возможности править оценку задачи**. Все часы ведутся в YouTrack — ответственность за оценку не размазывается между системами.

### Вопрос 7: Резерв времени — единый или индивидуальный

**Решение:** Резерв применяется **к каждому сотруднику индивидуально**.

Пример: если в месяце 168 рабочих часов, а резерв = 30%, то:
```
Доступные часы сотрудника = 168 - 168 × 0,30 = 117,6 часа
```

### Вопрос 8: Направления тестирования и управления — виртуальные или реальные люди

**Решение:**
- Часы тестирования и управления рассчитываются как процент от оценки задачи (п. 9.6 ТЗ).
- На этапе планирования мы **не можем чётко указать**, какие конкретные задачи тестирования какими сотрудниками будут делаться. Эти часы отображаются в столбцах направлений как суммарные.
- В СПО сотрудникам можно **выдавать роли** (разработчик, тестировщик, менеджер). Если у нас на тестировании, например, 3 сотрудника, то в столбце "Тестирование" нормальная мощность для планирования рассчитывается как `3 × 168 × 0,3`.
- **После факта** реальные часы тестирования и управления списываются конкретными сотрудниками, и эти данные подтягиваются из YouTrack.

### Вопрос 9: Уровень бизнес-группировки — глобальный или индивидуальный

**Решение:** Это **настройка для конкретного отчёта (периода)**. Пользователь может переключать уровень группировки в реальном времени и видеть, сколько задач решается на уровне Истории, сколько на уровне Фичи и т.д. (п. 11.3 ТЗ).

### Вопрос 10: Оценка бизнеса

**Решение:** Оценка бизнеса ставится **на задачу целиком**. Бизнесу не важно, сколько сотрудников по ней работало.

### Вопрос 11: Отображение сумм в личном отчёте до проставления оценок

**Решение:** Если оценки не проставлены — соответствующие суммы показываются как **0**. Никаких значений по умолчанию. Сотрудник видит нули в полях "Сумма от руководителя" и "Сумма от бизнеса", пока эти оценки не проставлены.

### Вопрос 12: Личный отчёт — план или факт

**Решение:**
- В личном отчёте показываем задачи, которые **были запланированы на сотрудника**, даже если в течение спринта он поменялся задачами с коллегами.
- В личном отчёте помимо затраченных часов должна отображаться **оценка** по задаче (чтобы руководитель при выставлении оценки видел этот результат).

### Вопрос 13: Ожидаемый объём данных

**Решение:** При планировании ожидается **менее 1000 задач в месяц**.

### Вопрос 14: Рабочие часы в году

**Решение:** Количество рабочих часов в году — это **одно значение на всех сотрудников** (общая настройка для периода/года).
