# Техническое задание v2
# Система Планирования и Отчетности (СПО)

**Версия:** 2.0  
**Дата:** 2026-04-25  
**Статус:** версия 2.0 — актуализирована после исследования YouTrack (аддендум v1.1)

---

## 1. Общая информация

СПО — веб-портал для планирования месячного спринта, контроля загрузки сотрудников, анализа план/факт и расчета финансовых показателей на основе данных из YouTrack.

YouTrack остается основной учетной системой разработки: в нем создаются задачи, ведутся статусы, оценки, связи задач, исполнители и фактически потраченное время. СПО не заменяет YouTrack, а использует его данные для управленческого планирования, финансовой оценки и отчетности.

СПО разворачивается как отдельное веб-приложение на отдельном сервере с собственной базой данных.

---

## 2. Цели системы

Основные цели СПО:

1. Планировать календарный месячный спринт с учетом доступной рабочей мощности сотрудников.
2. Исключить перегруз сотрудников, например ситуацию, когда при 168 рабочих часах на сотрудника запланировано 300 часов.
3. Учитывать резерв времени на внеплановые задачи месяца.
4. Фиксировать утвержденный план спринта.
5. Передавать признаки планирования обратно в YouTrack.
6. Загружать фактические результаты из YouTrack по итогам месяца.
7. Сравнивать план и факт.
8. Формировать итоговый отчет по месяцу для бизнеса, менеджеров, руководителей и директора.
9. Обеспечивать выставление оценок руководителя и бизнеса.
10. Рассчитывать личные отчеты сотрудников и финансовые показатели по задачам.
11. Хранить историю изменений планов, ставок, формул, оценок и закрытых отчетных периодов.

## 5. Роли и права доступа

Один пользователь может иметь несколько ролей одновременно.

### 5.1 Роли

| Роль | Описание |
|---|---|
| Пользователь | Видит только свой личный отчет по итогам месяца и свои ставки. Не может менять ставки и оценки. |
| Руководитель | Видит личные отчеты своих сотрудников и себя. Может выставлять оценку руководителя по каждой задаче своих сотрудников. |
| Менеджер | Видит отчеты руководителей и сотрудников. Может редактировать процент готовности задач. Может менять ставки сотрудников. Сам также может быть сотрудником и иметь личный отчет. |
| Директор | Видит все отчеты. Может менять ставки всех сотрудников, финансовые настройки, формулы, оценки руководителя и оценки бизнеса. Может переоткрывать закрытый период. |
| Бизнес | Видит итоговый отчет результатов работы всей группы разработки. Может выставлять оценку бизнеса по задачам. |
| Бухгалтер | Настраивает налоговые формулы (НДФЛ, страховые взносы, резерв отпускных и др.) через интерфейс администрирования. Не имеет доступа к операционным данным планирования. |
| Администратор | Администрирует систему, пользователей, роли, настройки интеграций. Не имеет доступа к финансовым отчетам, если у него нет дополнительной роли, например Директор. |

### 5.2 Принципы доступа

1. Руководитель может выставлять оценки только по своим сотрудникам.
2. Директор может корректировать любые оценки.
3. Менеджер может редактировать процент готовности задач, но не может выставлять оценку бизнеса.
4. Бизнес выставляет только оценку бизнеса по задаче целиком.
5. Администратор управляет настройками системы, но не видит финансы без дополнительной финансовой роли.
6. Бухгалтер управляет налоговыми формулами, но не видит операционные данные.
7. Все действия по изменению планов, ставок, формул, оценок и закрытых периодов логируются.

## 6. Интеграция с YouTrack и Hub

### 6.1 Назначение интеграции

Интеграция обеспечивает:

1. Получение задач из YouTrack.
2. Получение справочников и пользователей из YouTrack/Hub.
3. Получение статусов, оценок, фактически потраченного времени и связей задач.
4. Передачу признаков планирования обратно в YouTrack.
5. Синхронизацию пользователей и их статусов.

### 6.2 Параметры подключения

В административном интерфейсе должны настраиваться:

| Параметр | Обязательность | Описание |
|---|---:|---|
| Base URL YouTrack | Да | Базовый URL инстанса YouTrack, например `https://youtrack.company.local`. |
| API token | Да | Permanent token или иной токен доступа к YouTrack REST API. |
| Hub API URL | Опционально | URL Hub, если данные пользователей загружаются отдельно из Hub. |
| Hub API token | Опционально | Токен для чтения пользователей Hub. |
| Список проектов | Да | Проекты YouTrack, из которых загружаются задачи. |
| Query для незавершенных задач | Да | Поисковый запрос YouTrack для отбора незавершенных задач. |
| Agile board / sprint mapping | Да | Настройка доски/спринта для выгрузки плана. |
| Частота синхронизации | Да | Cron-like настройка или выбор интервала. |
| Ручной запуск синхронизации | Да | Кнопка запуска синхронизации вручную. |
| Timeout API-запросов | Да | Таймаут запросов к YouTrack. |
| Размер пачки синхронизации | Да | Количество задач в одном пакете. |
| Retry policy | Да | Количество повторов при ошибках. |
| Email для ошибок синхронизации | Да | Получатели уведомлений об ошибках. |

### 6.3 Маппинг полей YouTrack

В настройках интеграции должен быть интерфейс сопоставления полей YouTrack с полями СПО.

На основании исследования YouTrack (аддендум v1.1) подтверждены следующие фактические названия и типы полей:

| Поле СПО | YouTrack custom field | Тип в YouTrack | Тип данных |
|---|---|---|---|
| Номер задачи | `idReadable` | встроенное | строка |
| Ссылка на задачу | Формируется из Base URL + номер задачи | — | URL |
| Название задачи | `summary` | встроенное | строка |
| Проект | Project | встроенное | проект |
| Система | `Система` | перечисление[1] (Enum) | одиночный выбор |
| Спринт | `Спринт` (Enum) или Agile/Sprints API | перечисление[1] / Agile | одиночный выбор |
| Тип | `Type` | перечисление[1] (Enum) | одиночный выбор |
| Приоритет | `Priority` | перечисление[1] (Enum) | одиночный выбор |
| Статус | `State` | состояние[1] (State) | одиночный выбор |
| Постановщик | Reporter (встроенный) / `Постановщик` (кастомный) | пользователь / Multi-user | одиночный / множественный |
| Исполнитель | `Assignee` | пользователь[*] (Multi-user) | множественный выбор |
| Оценка | `Оценка` | период (Period) | длительность в минутах |
| Затраченное время | `Затраченное время` | период (Period) | длительность в минутах |
| Родительская задача | Связи `Parent for` / `Subtask of` | встроенное | связь |

Поле `Подсистема` (Enum, 27 значений) не используется в маппинге — система оперирует только признаком `Система`.

Фактические значения приоритетов (английские): `Blocker`, `High`, `Medium`, `Low`. В интерфейсе СПО допустимо отображать русские локализованные названия, но при сопоставлении с полем YouTrack используются английские значения.

Формат поля оценки: `PeriodProjectCustomField`, значение хранится в минутах (поле `minutes` внутри `PeriodValue`). Пример: `{"minutes": 120000, "presentation": "50н"}`. Конвертация в часы: `minutes / 60`.

### 6.4 Данные, получаемые при синхронизации задач

СПО должно загружать:

1. Все незавершенные задачи по настроенному запросу.
2. Задачи, по которым были списания времени в отчетном месяце.
3. Задачи из зафиксированного плана.
4. Связи родитель/дочерняя задача.
5. Work items за период.
6. Статусы задач.
7. Оценки задач.
8. Проект, систему, тип, приоритет, постановщика, исполнителя.

Задачи выгружаются в СПО независимо от того, заполнена в них оценка в YouTrack или нет. СПО не даёт возможности править оценку задачи — все часы ведутся в YouTrack, ответственность за оценку не размазывается между системами.

При синхронизации задач с несколькими исполнителями (Multi-user Assignee) необходимо оставлять только одного исполнителя (первого по списку).

### 6.5 Передача плана в YouTrack

После фиксации плана СПО должно передать в YouTrack:

1. Назначение основного исполнителя задачи (один исполнитель — тот, на кого запланирована задача).
2. Добавление задачи в sprint/agile board (через Agile/Sprints API).
3. Запись планового спринта в custom field (Enum-поле `Спринт`).
4. Запись плановых часов в custom field.
5. Добавление тега, например `SPO planned`.

На первой версии не требуется:

1. Добавлять комментарий с деталями планирования.
2. Менять статус задачи.

### 6.6 Журнал синхронизации

Для каждой синхронизации необходимо хранить:

- дата и время запуска;
- тип запуска: ручной/автоматический;
- инициатор запуска;
- статус: успешно, частично успешно, ошибка;
- количество обработанных задач;
- количество созданных/обновленных записей;
- ошибки по API;
- длительность выполнения;
- ссылка на детальный лог.

## 7. Импорт и управление пользователями

### 7.1 Импорт пользователей

Пользователи СПО импортируются из YouTrack/Hub.

Требования:

1. В администрировании должна быть кнопка синхронизации пользователей с YouTrack/Hub.
2. Новые пользователи из YouTrack/Hub создаются в СПО автоматически.
3. Если пользователь удален или заблокирован в YouTrack/Hub, в СПО он не удаляется автоматически, а помечается как `Уволен` или `Неактивен`.
4. По неактивным пользователям должна сохраняться возможность формировать отчеты.
5. Администратор СПО может удалить пользователя полностью вручную.
6. Все пользователи импортируются как отдельные учётные записи, даже если у одного сотрудника несколько логинов в YouTrack. Задача разрешения дублей возлагается на заказчика.

### 7.2 Данные пользователя

Для каждого пользователя должны храниться:

- ФИО;
- email;
- логин YouTrack;
- ID пользователя YouTrack/Hub;
- AD login;
- статус: активен, уволен, заблокирован, удален вручную;
- роли СПО;
- руководитель;
- рабочая роль для классификации трудозатрат: разработка, тестирование, управление, другое;
- история ставок;
- дата начала работы;
- дата увольнения, если применимо.

Маппинг полей из YouTrack API:
- `login` → логин в системе
- `name` / `fullName` → отображаемое имя
- `email` → email для уведомлений
- `banned` → признак блокировки (исключать из активных пользователей)

### 7.3 Авторизация через AD

На основании исследования YouTrack: Hub API недоступен как OAuth/OIDC-провайдер (ответ 404/401). Для аутентификации потребуется прямой LDAP/LDAPS к Active Directory.

СПО должно поддерживать:

1. Прямую авторизацию через LDAP/LDAPS к локальному AD.
2. Ручную связку учетной записи СПО с AD login при невозможности автоматического сопоставления.

## 8. Справочники и настройки системы

### 8.1 Основные справочники

СПО должно содержать следующие справочники:

1. Пользователи.
2. Роли доступа.
3. Рабочие роли сотрудников: разработка, тестирование, управление, другое.
4. Ставки сотрудников.
5. Проекты YouTrack.
6. Системы из YouTrack.
7. Типы задач.
8. Приоритеты.
9. Статусы.
10. Оценки бизнеса.
11. Оценки руководителя.
12. Формулы расчета.
13. Уведомления.

### 8.2 Настройки планирования

Настройки планирования:

| Настройка | Значение по умолчанию | Описание |
|---|---:|---|
| Длительность спринта | 1 календарный месяц | Фиксировано для первой версии. |
| Рабочие часы месяца | задается вручную | Количество рабочих часов в конкретном месяце. |
| Резерв на внеплановые задачи | 30% | Процент доступных часов, резервируемый под внеплановые работы. Применяется к каждому сотруднику индивидуально. |
| Процент тестирования | задается директором/админом | Процент от оценки разработки. |
| Процент отладки | задается директором/админом | Процент от оценки разработки; добавляется к часам разработчика. |
| Процент управления | задается директором/админом | Процент от оценки разработки. |
| Порог желтой зоны загрузки | настраивается | Например 80% от доступной мощности. |
| Порог красной зоны загрузки | настраивается | Например 100% от доступной мощности. |
| Уровень группировки для бизнес-отчета | История | Может быть: Эпик, Фича, История, Задача. Переключается в реальном времени для конкретного отчёта. |

### 8.3 Финансовые настройки

Редактируются ролями Директор и Бухгалтер.

Настройки должны версионироваться. При закрытии отчетного периода в отчете фиксируются все ставки, проценты, коэффициенты и формулы, действовавшие на момент закрытия/расчета. Последующие изменения настроек не должны пересчитывать закрытые отчеты.

## 9. Модуль планирования

### 9.1 Назначение

Модуль планирования предназначен для формирования плана календарного месячного спринта на основе незавершенных задач из YouTrack.

### 9.2 Подготовка периода планирования

При создании нового периода пользователь указывает:

1. Месяц и год.
2. Количество рабочих часов в месяце.
3. Резерв времени на внеплановые задачи, по умолчанию 30%.
4. Список сотрудников, участвующих в планировании.
5. Проценты тестирования, отладки и управления, если нужно переопределить общие настройки.
6. Проекты, системы и приоритеты, участвующие в планировании.

### 9.3 Расчет доступной мощности сотрудника

Для каждого сотрудника рассчитывается (резерв применяется к каждому сотруднику индивидуально):

```text
Доступные часы = Рабочие часы месяца × (1 - Резерв на внеплановые задачи)
```

Например:

```text
Рабочие часы месяца = 168
Резерв = 30%
Доступно для планирования = 168 × 70% = 117,6 часа
```

Если у сотрудника рабочая роль тестирования, то его нормальная мощность для столбца тестирования = количество тестировщиков × 168 × (1 - резерв).

### 9.4 Интерфейс планирования

Основной экран планирования должен быть представлен одной рабочей таблицей:

- слева — бэклог задач;
- далее — столбцы сотрудников;
- далее — столбцы направлений тестирования и управления;
- в верхней части каждого столбца — сумма часов;
- задачи можно перетаскивать из бэклога на сотрудников;
- задачи должны быть кликабельными и вести в YouTrack;
- задачи должны отображаться с номером, названием, типом, приоритетом, проектом, системой, статусом, оценкой и готовностью;
- задачи должны отображаться в древовидной структуре согласно связям `Parent for` / `Subtask of`.

### 9.5 Сортировка и подсветка бэклога

Задачи в бэклоге сортируются:

1. По степени завершенности из предыдущего месяца.
2. По приоритету.
3. По проекту/системе/типу, если включена соответствующая сортировка.

Для новых задач степень готовности равна 0%.

Приоритетные задачи подсвечиваются красной рамкой, если поле `Приоритет` равно:

- `Blocker`;
- `High`.

Запрета на планирование низкоприоритетных задач нет; приоритет используется для сортировки и визуальной подсветки.

### 9.6 Расчет часов при планировании задачи

При назначении задачи разработчику:

```text
Часы разработчика = Оценка задачи + Часы отладки
Часы отладки = Оценка задачи × % отладки
```

Тестирование и управление считаются отдельно по направлению:

```text
Часы тестирования = Оценка задачи × % тестирования
Часы управления = Оценка задачи × % управления
```

Пример:

```text
Оценка задачи = 10 часов
Отладка = 30%
Тестирование = 20%
Управление = 10%

Сотрудник-разработчик = 10 + 3 = 13 часов
Тестирование = 2 часа
Управление = 1 час
```

### 9.7 Цветовая дифференциация загрузки

Для каждого сотрудника отображается суммарная загрузка.

Цветовая индикация:

| Зона | Условие | Цвет |
|---|---|---|
| Норма | Загрузка ниже желтого порога | Зеленый/нейтральный |
| Близко к пределу | Загрузка между желтым и красным порогом | Желтый |
| Перегруз | Загрузка выше красного порога | Красный |

Пороги должны настраиваться.

### 9.8 Фильтры планирования

Должны быть фильтры:

- система;
- проект;
- приоритет;
- тип задачи;
- статус;
- исполнитель;
- постановщик;
- запланирована/не запланирована;
- степень готовности;
- текстовый поиск по номеру и названию задачи.

### 9.9 Фиксация плана

После формирования план фиксируется.

При фиксации:

1. Создается неизменяемая базовая версия плана.
2. Фиксируются список задач, сотрудники, плановые часы, проценты, настройки и расчетные значения.
3. План выгружается в YouTrack.
4. Дальнейшие изменения доступны только роли Директор.
5. Все изменения после фиксации записываются в журнал изменений.

---

## 10. Изменение зафиксированного плана

После фиксации плана изменения может вносить только Директор.

Каждое изменение должно логироваться:

- дата и время;
- пользователь;
- сущность изменения;
- старое значение;
- новое значение;
- причина изменения, если указана;
- период;
- задача, если изменение относится к задаче.

Журнал изменений должен быть доступен:

1. Из карточки периода.
2. Из экрана планирования.
3. Из итогового отчета месяца.
4. В экспорте.

## 11. Иерархия задач и группировка

### 11.1 Иерархия YouTrack

В YouTrack используется вложенность:

```text
Эпик → Фича → История → Задача/Ошибка
```

Связи строятся через `Parent for` / `Subtask of`.

### 11.2 Отображение дерева

В планировании и отчетах задачи должны отображаться древовидно.

Если дочерняя задача попала в план или факт, а родительская нет, родитель должен отображаться как контекстная строка без собственных часов.

При формировании бэклога показываются задачи не выше уровня "История". Если у задачи или ошибки (бага) нет родителя-Истории, она показывается как есть. Если у Истории есть родитель (эпик или фича), то при наведении на неё отображается tooltip с полным названием задачи и номером тикета родителя.

### 11.3 Настраиваемый уровень бизнес-группировки

В настройках отчёта (периода) должен задаваться уровень отображаемой группировки:

- Эпик;
- Фича;
- История;
- Задача.

По умолчанию: `История`.

Настройка переключается в реальном времени для конкретного отчёта (периода). Пользователь может переключать уровень группировки и видеть, сколько задач решается на уровне Истории, сколько на уровне Фичи и т.д.

Если выбран уровень `История`, то в бизнес-отчете отображается история, а часы и суммы агрегируются по всем вложенным задачам и ошибкам.

Если у задачи или ошибки нет родителя выбранного уровня, она отображается как есть.

## 12. Итоговый отчет периода

### 12.1 Назначение

Итоговый отчет формируется по завершении месяца после загрузки факта из YouTrack.

Отчет должен включать:

1. Все задачи зафиксированного плана.
2. Все задачи, по которым были списания времени в отчетном периоде.
3. Признак запланированности.
4. Возможность фильтрации по признаку запланированности.
5. Суммы и проценты, пересчитываемые с учетом примененных фильтров.

### 12.2 Поля итогового отчета

| Поле | Источник/расчет |
|---|---|
| Номер задачи | YouTrack |
| Ссылка на задачу | YouTrack URL |
| Название задачи | YouTrack |
| Родительская задача | YouTrack links |
| Проект | YouTrack |
| Система | YouTrack custom field |
| Тип | YouTrack custom field |
| Приоритет | YouTrack custom field |
| Статус | YouTrack |
| Постановщик | YouTrack |
| Основной исполнитель | Первый исполнитель из YouTrack или назначенный в плане |
| Признак запланированности | СПО |
| Плановые часы разработки | СПО / YouTrack plan field |
| Плановые часы тестирования | СПО |
| Плановые часы управления | СПО |
| Фактически потраченные часы | YouTrack work items |
| Часы разработки факт | Расчет по роли сотрудника |
| Часы тестирования факт | Расчет по роли сотрудника |
| Часы управления факт | Расчет по роли сотрудника |
| Статус задачи | YouTrack |
| Осталось часов | Плановая оценка - фактически потрачено |
| % готовности | СПО, редактируют Директор и Менеджер |
| Себестоимость плановая | Расчет по формуле |
| Себестоимость фактическая | Расчет по формуле |
| Оставшиеся затраты | Расчет по формуле |
| Оценка бизнеса | СПО |
| Комментарий | СПО, опционально |

### 12.3 Расчет оставшихся часов

```text
Осталось часов = Плановая оценка - Фактически потрачено
```

Если значение отрицательное:

1. Значение подсвечивается.
2. В строке задачи отображается отрицательное отклонение.
3. В итоговых суммах отрицательные часы и рубли не учитываются как отрицательные значения, чтобы не искажать картину.

### 12.4 Группировка и итоги

Все задачи группируются по признаку `Система`.

Для каждой системы отображаются суммирующие значения по всем числовым столбцам.

Также должны поддерживаться группировки:

- по проекту;
- по системе;
- по типу;
- по приоритету;
- по запланированности;
- по уровню бизнес-группировки: история/фича/эпик.

### 12.5 Сортировка и фильтрация

По любому полю итоговой таблицы должны быть доступны:

- сортировка;
- фильтрация;
- текстовый поиск;
- сохранение пользовательских фильтров, если возможно.

### 12.6 Статистика выполнения плана

В итоговом отчете должны отображаться показатели:

| Показатель | Формула |
|---|---|
| Плановые часы | Сумма плановых часов по фильтру |
| Фактические часы | Сумма фактических часов по фильтру |
| Отклонение часов | Факт - План |
| % выполнения плана | Факт / План × 100% |
| Часы вне плана | Фактические часы по незапланированным задачам |
| % внеплановой нагрузки | Внеплановые часы / Все фактические часы × 100% |
| Оставшиеся часы | Сумма положительных значений остатка |
| Незавершенные плановые задачи | Количество плановых задач, не завершенных к концу периода |

## 13. Оценки руководителя и бизнеса

### 13.1 Оценка руководителя

Оценка руководителя выставляется по связке:

```text
Сотрудник + Задача + Период
```

Если над одной задачей работали несколько сотрудников, задача отображается в личном отчете каждого сотрудника, и руководитель выставляет каждому отдельную оценку.

Оценки руководителя:

| Оценка | Процент |
|---|---:|
| Плохо | 0% |
| Удовлетворительно | 10% |
| Хорошо | 20% |
| Отлично | 30% |

Значение по умолчанию: `Хорошо`.

При выставлении оценки руководителя можно оставить опциональный комментарий (текстовое поле). Комментарий отображается в итоговом отчёте и личном отчёте сотрудника при наличии.

### 13.2 Оценка бизнеса

Оценка бизнеса выставляется по задаче целиком в итоговом отчете. Бизнесу не важно, сколько сотрудников по ней работало.

Бизнес не видит детализацию по сотрудникам и не оценивает вклад каждого сотрудника.

Оценки бизнеса:

| Оценка | Процент |
|---|---:|
| Нет пользы | 0% |
| Прямая выгода | 10% |
| Польза очевидна | 20% |

Значение по умолчанию: `Прямая выгода`.

При выставлении оценки бизнеса можно оставить опциональный комментарий (текстовое поле). Комментарий отображается в итоговом отчёте и личном отчёте сотрудника при наличии.

### 13.3 Трансляция оценки бизнеса в личные отчеты

После выставления оценки бизнеса по задаче эта оценка автоматически отображается в личных отчетах всех сотрудников, которые работали по этой задаче в отчетном периоде.

### 13.4 Изменение оценок после закрытия периода

После закрытия периода оценки не могут изменяться обычными ролями.

Только Директор может переоткрыть период, внести правку и закрыть его повторно.

Все изменения фиксируются в журнале изменений.

## 14. Расчет зарплаты и эффективной ставки

### 14.1 Базовая часовая ставка сотрудника

Для сотрудника задается согласованная сумма ЗП на руки в месяц.

На основе нее рассчитывается базовая часовая ставка:

```text
Годовой доход на руки = ЗП на руки в месяц × 12
Базовая часовая ставка = Годовой доход на руки / Количество рабочих часов в году
```

Количество рабочих часов в году задается в настройках года. Это единое значение на всех сотрудников.

### 14.2 История ставок

Для каждого сотрудника должна храниться история ставок:

- дата начала действия;
- дата окончания действия;
- месячная сумма на руки;
- количество рабочих часов в году;
- рассчитанная часовая ставка;
- кто создал/изменил;
- дата изменения;
- комментарий.

Если ЗП сотрудника меняется в течение года, создается новая ставка с датой начала действия. Ставка применяется по расчетному месяцу.

Ставки могут редактировать только Директор и Менеджер.

Пользователь видит свою ставку в личном кабинете, но не может ее менять.

### 14.3 Принцип расчета по задаче

Все денежные показатели рассчитываются от базовой часовой ставки сотрудника и фактически потраченных часов.

Оценки руководителя и бизнеса формируют премиальную часть и эффективную ставку.

### 14.4 Настраиваемые параметры расчета

Роль Директор должна иметь возможность редактировать:

- базовый процент;
- правила премиальной части;
- проценты оценок руководителя;
- проценты оценок бизнеса;
- формулы эффективной ставки;
- формулы плановой и фактической себестоимости.

Роль Бухгалтер должна иметь возможность редактировать:

- формулы НДФЛ;
- формулы страховых взносов;
- формулы резерва отпускных;
- формулу `Итого с налогами`.

В первой версии используется безопасный конфигуратор без произвольного кода: директор и бухгалтер меняют проценты и выбирают преднастроенные формулы.

### 14.5 Расчет суммы на руки по задаче

Базовая формула:

```text
Сумма базовая = Часы × Базовая часовая ставка × Базовый процент
Сумма от руководителя = Часы × Базовая часовая ставка × Процент оценки руководителя
Сумма от бизнеса = Часы × Базовая часовая ставка × Процент оценки бизнеса
Итого на руки = Сумма базовая + Сумма от руководителя + Сумма от бизнеса
```

При настройках:

```text
Базовый процент = 70%
Оценка руководителя Хорошо = 20%
Оценка бизнеса Прямая выгода = 10%
```

сотрудник по умолчанию получает 100% базовой расчетной суммы.

При максимальных оценках:

```text
Базовый процент = 70%
Оценка руководителя Отлично = 30%
Оценка бизнеса Польза очевидна = 20%
```

сотрудник получает 120% от базовой расчетной суммы, то есть премию +20%.

Если оценки не проставлены — соответствующие суммы показываются как 0. Никаких значений по умолчанию для расчёта.

### 14.6 Налоги и дополнительные начисления

В личном отчете должны рассчитываться:

- НДФЛ;
- страховые взносы;
- резерв отпускных;
- итого с налогами.

Эти значения рассчитываются по формулам от суммы `Итого на руки по задаче`. Формулы и проценты задает Бухгалтер в настройках системы.

### 14.7 Эффективная ставка

```text
Эффективная ставка = Итого на руки по задаче / Часы по задаче
```

Эффективная ставка используется для расчета себестоимости задачи.

## 15. Себестоимость

### 15.1 Фактическая себестоимость

Фактическая себестоимость задачи рассчитывается на основе:

- фактических часов сотрудников;
- базовых ставок сотрудников за расчетный месяц;
- оценок руководителя по каждому сотруднику и задаче;
- оценки бизнеса по задаче;
- эффективных ставок сотрудников;
- налоговых и дополнительных формул, если они включены в конфигурации.

### 15.2 Плановая себестоимость

Плановая себестоимость рассчитывается на основе:

- плановых часов;
- базовых ставок назначенных сотрудников;
- оценок по умолчанию:
  - бизнес: `Прямая выгода` = 10%;
  - руководитель: `Хорошо` = 20%;
- эффективной ставки, рассчитанной по значениям по умолчанию.

### 15.3 Оставшиеся затраты

Оставшиеся затраты рассчитываются по настраиваемой формуле.

Базовая формула:

```text
Оставшиеся затраты = Положительные оставшиеся часы × Плановая эффективная ставка
```

Если оставшиеся часы отрицательные, в итоговых суммах они учитываются как 0.

## 16. Личный кабинет сотрудника

### 16.1 Назначение

Личный кабинет показывает сотруднику отчет за месяц по всем задачам, которые:

1. Планировались на сотрудника (даже если в течение спринта он поменялся задачами с коллегами).
2. Фактически выполнялись сотрудником.

### 16.2 Поля личного отчета

В личном отчете должны отображаться:

| Поле | Описание |
|---|---|
| Номер задачи | Кликабельная ссылка на YouTrack |
| Название задачи | Кликабельная ссылка на YouTrack |
| Статус задачи | Из YouTrack |
| Номер родительской задачи | Из YouTrack |
| Оценка задачи | Оценка из YouTrack (в часах) |
| Часы | Фактически потраченные часы сотрудника |
| Сумма постоянная | На основе базового процента |
| % от руководителя | Оценка руководителя |
| Сумма от руководителя | Расчетная сумма |
| % от бизнеса | Оценка бизнеса |
| Сумма от бизнеса | Расчетная сумма |
| Итого на руки | Итог по задаче |
| НДФЛ | По формуле |
| Страховые взносы | По формуле |
| Резерв отпускные | По формуле |
| Итого с налогами | По формуле |
| Эффективная ставка | Итого на руки / часы |

Вверху отчета под заголовком должны отображаться суммирующие итоги по каждому числовому столбцу.

### 16.3 Актуализация личных отчетов

Личные отчеты формируются сразу после загрузки факта.

До закрытия периода они автоматически актуализируются при:

- выставлении оценки руководителя;
- выставлении оценки бизнеса;
- изменении процента готовности;
- пересчете факта;
- изменении данных периода.

После закрытия периода личные отчеты замораживаются и не пересчитываются, кроме сценария переоткрытия периода Директором.

## 17. Workflow отчетного периода

Период проходит следующие состояния:

```text
1. Планирование
2. План зафиксирован
3. Месяц в работе
4. Факт загружен
5. Руководители выставляют оценки
6. Бизнес выставляет оценки
7. Директор проверяет
8. Период закрыт
9. Период переоткрыт директором, если требуется корректировка
10. Период повторно закрыт
```

### 17.1 Закрытие периода

При закрытии периода фиксируются:

- план;
- факт;
- ставки;
- проценты;
- формулы;
- оценки руководителей;
- оценки бизнеса;
- расчеты личных отчетов;
- итоговый отчет;
- финансовые показатели;
- журнал изменений.

После закрытия период не пересчитывается автоматически.

### 17.2 Переоткрытие периода

Только Директор может переоткрыть закрытый период.

При переоткрытии указывается причина.

Все изменения после переоткрытия фиксируются в журнале.

## 18. Классификация фактических часов

Так как в YouTrack нет признака роли сотрудника в work item, СПО классифицирует фактические часы на основе рабочей роли сотрудника в СПО.

Пример:

| Рабочая роль сотрудника | Фактические часы попадают в направление |
|---|---|
| Разработчик | Разработка |
| Тестировщик | Тестирование |
| Руководитель / Менеджер | Управление |
| Другое | Другое / требуется настройка |

В итоговом отчете должно сравниваться:

- план тестирования по направлению vs фактические часы сотрудников с ролью тестирования;
- план управления по направлению vs фактические часы сотрудников с ролью управления;
- план разработки vs фактические часы сотрудников с ролью разработки.

Фактические часы тестировщиков и руководителей также попадают в их личные отчеты и участвуют в финансовом расчете.

## 24. Нефункциональные требования

### 24.1 Производительность

1. Таблицы планирования и отчетов должны работать с тысячами задач без заметных задержек (ожидаемый объём — менее 1000 задач на месяц).

3. Пользователь должен видеть статус выполнения длительных операций.

1. Ошибки синхронизации не должны приводить к потере ранее загруженных данных.

4. Настройки и формулы должны версионироваться.

### 24.3 Безопасность

1. Доступ к данным ограничивается ролями.
2. Финансовые данные доступны только разрешенным ролям.
3. API-токены YouTrack/Hub должны храниться в зашифрованном виде.
4. Должен быть аудит входов и критичных действий.
5. Должна быть защита от несанкционированного изменения закрытых отчетов.

### 24.4 Резервное копирование

Должны быть предусмотрены:

- резервные копии PostgreSQL;
- хранение минимум нескольких последних бэкапов;
- инструкция восстановления;
- резервное копирование конфигураций и журналов.

## 25. API и интеграционные замечания по YouTrack

Для интеграции должны использоваться публичные REST API YouTrack и Hub.

Ключевые области API:

1. Issues API — получение задач, полей, статусов, связей, исполнителей. Пример запроса: `GET /api/issues?fields=idReadable,summary,customFields(id,name,value($type,name,minutes))`.
2. Custom fields API — чтение и запись значений кастомных полей.
3. Work items API — получение фактически потраченного времени.
4. Agile/Sprints API — работа со спринтами и привязкой задач к спринтам.
5. Users/Hub API — синхронизация пользователей. Поля: `id`, `login`, `name`, `fullName`, `email`, `banned`.
6. Permanent token authorization — авторизация серверного приложения.

**Подтверждённые эндпоинты (по результатам исследования):**

- `GET /api/issues/{issueId}?fields=idReadable,summary,customFields(id,name,value($type,name,login,minutes,presentation))` — получение задачи со всеми полями
- `GET /api/admin/projects?fields=id,name,shortName,customFields(id,name,fieldType(presentation))` — проекты и их кастомные поля
- `GET /api/admin/projects/{projectId}/customFields/{fieldId}?fields=id,name,fieldType(presentation),bundle(id,name,values(id,name,isResolved))` — значения enum-полей
- `GET /api/users?fields=id,login,name,email,fullName,banned` — список пользователей
- `GET /api/agiles?fields=id,name,sprints(id,name,start,finish)` — список досок и спринтов
- `GET /api/admin/customFieldSettings/bundles/state?fields=id,name,values(id,name,isResolved)` — статусы с признаком завершённости

Ссылки для проверки реализации:

- https://www.jetbrains.com/help/youtrack/devportal/authentication-with-permanent-token.html
- https://www.jetbrains.com/help/youtrack/devportal/api-concept-custom-fields.html
- https://www.jetbrains.com/help/youtrack/devportal/resource-api-issues-issueID-sprints.html

## 27. Открытые вопросы (закрыты по результатам исследования)

На момент подготовки версии 1.0 были открыты вопросы, которые требовали уточнения. По результатам прямого исследования YouTrack (аддендум к ТЗ v1.1) все вопросы закрыты:

### 27.1 — Точные названия custom fields в YouTrack и их типы

**Статус: закрыто.** Подтверждены названия и типы: `Система` (Enum), `Спринт` (Enum), `Type` (Enum), `Priority` (Enum), `State` (State), `Assignee` (Multi-user), `Постановщик` (Multi-user), `Оценка` (Period), `Затраченное время` (Period). Детальная таблица маппинга — в разделе 6.3.

### 27.2 — Точный формат поля оценки в YouTrack

**Статус: закрыто.** Поле `Оценка` — тип `PeriodProjectCustomField`. Значение хранится в минутах (поле `minutes` внутри `PeriodValue`). Пример: `{"minutes": 120000, "presentation": "50н"}`. Конвертация в часы: `minutes / 60`.

### 27.3 — Как именно в YouTrack настроены sprint/agile board и custom field спринта

**Статус: закрыто.** Используются два подхода: (1) Agile/Sprints API — доски с именованными спринтами с датами (например, доска "Cloud" со спринтами вида `Спринт 16 [01.04-30.04]`); (2) Enum-поле `Спринт` с 32 значениями-названиями спринтов. Рекомендуется использовать Agile/Sprints API как основной источник, Enum-поле как fallback.

### 27.4 — Какой источник использовать для постановщика

**Статус: закрыто.** Reporter (встроенное поле) как основной, кастомное поле `Постановщик` (Multi-user) как расширенный/дополнительный.

### 27.5 — Точные значения статусов, которые считаются завершенными

**Статус: закрыто.** Завершённость определяется флагом `isResolved: true`. Основные завершённые статусы: `Завершено`, `Архив`. В проектах также могут быть `Done`, `Duplicate`, `Размещено в продакшн`, `Передача ТП/ОТК` и другие. Проверять по `isResolved`, а не по имени.

### 27.6 — Точные значения приоритетов

**Статус: закрыто.** Фактические значения: `Blocker`, `High`, `Medium`, `Low` (английские). В ТЗ версии 1.0 были ошибочно указаны русские `Блокер`/`Высокий`. В интерфейсе СПО допустима русская локализация, но маппинг с YouTrack — по английским значениям.

### 27.7 — Доступность Hub как OAuth/OIDC-провайдера

**Статус: закрыто.** Hub API недоступен (404/401). Для аутентификации требуется прямой LDAP/LDAPS к Active Directory.

### 27.8 — Налоговые формулы

**Статус: закрыто.** Настройка налоговых формул вынесена на уровень роли **Бухгалтер**. Формулы являются редактируемыми через интерфейс администрирования. Механизм настраиваемых процентов и преднастроенных формул (п. 14.4) покрывает потребность.

### 27.9 — Комментарии к оценкам

**Статус: закрыто.** Комментарии к оценкам необходимо хранить. Предусмотреть опциональное текстовое поле (nullable text) при выставлении оценки руководителя и оценки бизнеса.

### 27.10 — Интеграция с бухгалтерской системой

**Статус: закрыто.** Не входит в MVP. Заложить архитектурную возможность экспорта данных по закрытым периодам в JSON/CSV для последующей интеграции.

## 28. Критерии приемки верхнего уровня

Система считается готовой к приемке первой версии, если:

1. Администратор может подключить СПО к YouTrack и выполнить успешную синхронизацию.
2. В СПО отображаются незавершенные задачи из YouTrack с нужными полями.
3. Задачи отображаются кликабельными ссылками на YouTrack.
4. Задачи отображаются в иерархии согласно связям Parent/Subtask.
5. Пользователь может создать период планирования на календарный месяц.
6. Пользователь может указать рабочие часы месяца и резерв времени.
7. Пользователь может распределить задачи по сотрудникам через drag-and-drop.
8. СПО показывает загрузку сотрудников и подсвечивает перегруз.
9. План можно зафиксировать.
10. План выгружается в YouTrack согласно заданным правилам.
11. После окончания месяца СПО загружает факт из YouTrack.
12. Итоговый отчет показывает плановые и фактические задачи.
13. В итоговом отчете можно фильтровать, сортировать и группировать данные.
14. Бизнес может выставить оценку бизнеса по задаче.
15. Руководитель может выставить оценку сотруднику по задаче.
16. Личный отчет сотрудника рассчитывает суммы по задачам.
17. Закрытый период не пересчитывается после изменения глобальных настроек.
18. Директор может переоткрыть период, внести изменения и закрыть его повторно.
19. Все критичные изменения отображаются в журнале.
20. Экспорты формируются в заявленных форматах.
